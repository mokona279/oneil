# 전략 개정 정책 — 언제, 어떻게 바꾸는가

**작성**: 2026-07-19 · **적용 대상**: 규칙서 v4.0(`oneil_strategy_v4_draft.md`,
`config/rules_v3-3.yaml`) 및 이후 버전 · **지위**: 규칙 자체가 아니라 **규칙을
바꾸는 규칙**(거버넌스). 문헌 조사(§7 출처) + 본 레포 P0~P8 전례를 성문화했다.

> **§7 리스크 거버너(3연속 손절 → 10일 정지)·시장필터와의 구분**: 그것들은
> 규칙서 안의 **운영 중단** 규칙이고, 본 문서는 규칙서 밖의 **개정 판단** 규칙이다.
> 거버너 발동은 개정 사유가 아니다(설계된 정상 동작).

## §1 문헌 요약 — 왜 이렇게 정하는가

- **성과 저하는 기본값이다**: 공표된 전략 97개 실측에서 진짜 OOS 수익은 백테스트
  대비 평균 -26%, 공개 후 -58%. 다른 집계(355개 전략)에서 Sharpe는 IS→OOS 평균
  -33%. → **포워드가 백테스트보다 나쁜 것 자체는 개정 사유가 아니다.** 임계
  이탈만 사유다(§3.3).
- **중단·경보 기준은 분포로 잡는다**: 백테스트 MDD 1회 관측치가 아니라 시뮬레이션
  분포(몬테카를로 확률 콘)의 꼬리로 "정상 변동 vs 고장"을 가른다. 운용사 관행
  기준의 하나는 **최대 MDD의 1.5배 + 24개월 약정**(Davey 인용). 본 레포의 P8-2
  월 단위 시작점 앙상블(102런)이 몬테카를로의 실측 대체물이다.
- **재최적화는 정기 창구로**: 워크포워드 방법론(Pardo)의 본질은 "정해진 주기로
  재검증하고, 그 밖에는 손대지 않는다". 파라미터 수 최소화가 1차 과적합 방어.
- **시도 횟수가 곧 편향이다**(Bailey·López de Prado, PBO/DSR): 시도 3회만으로도
  가짜 유의 전략이 나온다. 100개 변형 중 최고 Sharpe 2.0의 실질은 0.5 수준일 수
  있다. → 스윕 시도 수 기록(현행 문화)·개정 빈도 상한·홀드아웃 재사용 계상.

## §2 개정 등급 — 무엇을 바꾸는가에 따라 절차가 다르다

| 등급 | 내용 | 절차 | 전례 |
|---|---|---|---|
| **L0 현행화** | 규칙 불변·환경 반영: 세율/수수료(costs.yaml), 데이터 재수집 | 확인 즉시 반영 + 골든 영향 병기. 승인만, 검증 배터리 불요 | 2026 거래세 인상(P7, -0.07%p 병기) |
| **L1 증거 재분류** | 규칙·값 불변, **근거 문서만** 수정(채택 근거의 강화/제외) | 실측 근거 + 사용자 승인. 버전 태그 불변 | P8-3(캡처 지표를 slots12·W1 근거에서 제외), P7 승인 반영 |
| **L2 파라미터 개정** | 기존 규칙의 값 변경 | §4 전체 절차 + 연차 창구 원칙 | P1(k=5)·P2(N=12)·Q13(슬롯 12) |
| **L3 구조 개정** | 규칙 추가/제거/재설계 | §4 전체 절차(최고 강도) + 연차 창구 원칙 | P4(R4b 재진입 신설), P3(핸들 기각) |

버전 태그: L2는 마이너 증가(v4.0→v4.1), L3는 사용자와 협의(구조 누적 시 메이저).
모든 승격은 "vN 재현은 키 X를 Y로" 재현 키를 규칙 파일 헤더에 병기(현행 관례).

## §3 개정 타이밍

### §3.1 정기 — 연차 재검증 창 (매년 1월, 전년도 데이터 확정 후)

유일한 L2/L3 정기 창구. 작업 목록(이미 약속된 존속 의무 포함):

1. 데이터 연장 수집 + 말단 이상 점검(P7 §5 방식: 재수집 대조·소급 수정주가)
2. 풀런 갱신 → 성적표를 기준 경로에 연결(절단 헤드라인 병기 관례: +755%/+550%)
3. **9암 walkforward_report 재실행** — 규칙서 존속 의무("F1 갱신마다") 이행.
   신규 연도 슬라이스의 축별 delta 부호 확인
4. **§3.3 추적 지표 집계**: R3b·confirm5·slots12·R4b(재진입) 발동 건수·손익
   기여 분리 집계 — 저표본 개정 축의 과적합 추적 의무
5. 캡처 세트 갱신 + 티어별 캡처율(절단 대조 포함, P8-3 방식)
6. 포워드 원장 연간 집계(§3.2 누적분) vs 백테스트 분포 대조
7. 판정: 전 축 존속 / L1 재분류 상정 / L2·L3 개정 상정 — `plan/` 기록 + 승인

### §3.2 분기 — 경량 점검 (재검증 아님, 30분 이내)

- `forward/status.md` 집계 확인: actionable 신호 수·시장필터 상태 분포,
  마감된 신호의 경과(피벗 대비 수익·손절 도달)
- §3.3 트리거 표와 대조만 — **임계 미달이면 아무것도 바꾸지 않는다**
- `git add forward` 봉인 커밋 누락 여부 확인

### §3.3 수시 — 트리거 표 (충족 시에만 창구 밖 검토 개시)

### §3.4 개정 금지 조건 (트리거보다 우선)

- **통증 기반 개정 금지**: 드로다운 중이라도 T1·T2 임계 미충족이면 손대지 않는다.
  §1 첫 항 — OOS 저하는 기대 범위다. 임계 충족 시에도 순서는 "중단 → 재검증 →
  상정"이지 "즉석 수정"이 아니다.
- **표본 미달 판정 금지**: 포워드 30트레이드 미만·축 발동 5건 미만(-P2·Q13에서
  저표본 기각 전례)은 어느 방향으로도 증거로 쓰지 않는다.
- **창구 밖 L2/L3 금지**: T4를 제외한 모든 값·구조 변경은 연차 창(§3.1)으로
  배칭한다. 검토는 언제든 가능하나 **반영**은 창구에서.

## §4 개정 방법 (L2·L3 절차 — P1~Q13에서 검증된 파이프라인의 성문화)

1. **가설 사전 등록**: `plan/` 문서에 변경안·예상 효과·판정 지표·**기각 조건**을
   실측 전에 기록(사후 합리화 차단). 전례: 각 Phase 문서 서두.
2. **구현은 기본 off·골든 불변**: 새 코드는 config 키 null/off가 기존과 비트
   동일해야 한다(전례: Q14 rank_top_pct 기본 off).
3. **아블레이션·스윕**: 축 분리 on/off, 스윕 채택값은 **플래토 중앙**(경계값
   금지 — R1 k=5·confirm5 전례). **시도한 조합 수를 문서에 기록**(§1 PBO).
4. **검증 배터리**(전부, 생략 금지): 연도 슬라이스 워크포워드 · 말단 절단 ·
   시작점 앙상블 · 비용 민감도(15/30/50bp) · PBO-lite 순위 상관 · 캡처 티어.
   P5~P7이 잣대의 정본.
5. **적대적 리뷰**: 반대 증거를 명시적으로 탐색(단일 레짐 의존·저표본·경로
   단일사건 여부 — P7 항목). 유리한 해석 금지, 애매하면 "판정 불가+표본 수".
6. **사용자 승인 → 버전 승격**: rulebook_version 증가, 재현 키 병기, 규칙서
   개정 이력 문단 추가, PROGRESS 갱신, 커밋.
7. **개정 후 리셋**: 포워드 원장은 rules_sha8로 신구 자동 구분(P8-1 설계) —
   **새 버전의 T1 표본 카운트는 0에서 다시 시작**한다. 이전 버전 기록은 보존.

## §5 과적합 방어 원칙 (전 등급 공통)

- **포워드 원장으로 튜닝 금지**: `forward/`는 검증 전용이다. 파라미터 선택에
  쓰는 순간 유일한 진짜 OOS가 소멸한다. 튜닝에 오염됐다면 그 시점 이후 기록만
  새 OOS로 인정.
- **홀드아웃 재사용 계상**: 절단 창·검증 창을 재사용할 때마다 그 사실을 기록
  (재사용 횟수만큼 증거력 할인 — P8-3이 "3개 독립 방법"을 명기한 이유).
- **동시 다발 변경 금지**: 창구당 축별 분리 실측 없는 패키지 채택 금지
  (패키지 성적은 P5처럼 축 분해를 병기).
- **개정 이력 총량 감시**: 연차 창마다 누적 개정 축 수·시도 수를 집계해 병기 —
  축이 늘수록 DSR 논리로 헤드라인 성과를 스스로 할인해 읽는다.

## §6 연차 재검증 실행 프롬프트 (매년 1월 새 세션에 붙여넣기)

```
규칙서 연차 재검증을 진행한다(개정 아님 — 개정은 결과가 상정할 때만).
먼저 읽어라: docs/strategy_revision_policy.md(본 정책 — §3.1 작업 목록),
docs/oneil_strategy_v4_draft.md 헤더(존속 의무), plan/p5_walkforward.md·
plan/p7_adversarial.md(잣대 정본), forward/status.md(포워드 누적).

작업: 정책 §3.1의 1~7을 순서대로. 데이터 연장 → 말단 이상 점검 → 풀런 갱신
→ 9암 walkforward_report 재실행 → §3.3 추적 지표(R3b·confirm5·slots12·R4b
발동·기여) → 캡처 갱신 → 포워드 vs 백테스트 대조 → 판정.

규율: 임계(§3.3 트리거 표) 미충족이면 "전 축 존속"으로 종결한다 — 개선 아이디어가
보여도 반영은 상정·승인 후. 유리한 해석 금지. 저표본(축 발동 5건 미만·포워드
30건 미만)은 증거로 쓰지 않는다. 결과는 plan/annual_<연도>.md에 기록하고
PROGRESS 갱신·커밋. 장시간 런은 분리 프로세스(RAM 7.4GB, 동시 실행 금지).
```
