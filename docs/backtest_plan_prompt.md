# 주도주 매매규칙서 백테스트 — 구현 계획서 작성 프롬프트

> **사용법**: 이 프롬프트 전문을 새 세션에 붙여넣고, 매매규칙서 v3.2 원문을 반드시 함께 첨부한다. 규칙서 없이 실행하면 아래 요약만으로 계획이 작성되므로 정확도가 떨어진다.

---

## 1. 역할과 산출물

너는 퀀트 백테스트 시스템을 설계해 본 파이썬 아키텍트다. 아래 요구사항을 바탕으로 **구현 계획서**를 작성하라.

- 이번 산출물은 계획서다. **코드는 작성하지 않는다.** 단, 핵심 인터페이스의 시그니처(클래스명, 메서드명, 인자, 반환 타입)는 의사코드 수준으로 포함한다.
- 계획서의 각 단계는 이후 **개별 세션에서 독립적으로 구현**할 수 있는 크기여야 한다. 각 세션은 선행 단계가 남긴 파일과 테스트만으로 착수 가능해야 한다.
- 첨부된 매매규칙서 v3.2가 규칙의 단일 진실 원천이다. 아래 §3 요약과 규칙서가 다르면 규칙서를 따르고, 차이점은 질문 목록에 기록하라.

## 2. 목적

- 규칙서를 기계적으로 실행했을 때의 수익률·리스크 특성 검증
- 규칙 간 상호작용(예: 과열 제외 vs 베이스 판별, 손절 캡 vs 포지션 사이징) 점검
- 이후 파라미터 민감도 분석이 가능한 구조 확보

## 3. 구현 대상 규칙 범주 (참고 요약 — 최종 수치는 규칙서 확인 필수)

1. **Stage 0 추세 템플릿 필터**: 이동평균 배열, 52주 가격 위치 (미너비니 트렌드 템플릿)
2. **베이스 판별기**: 깊이 티어 — D ≤ 15% → 최소 5주, 15% < D ≤ 33% → 최소 7주, D > 33% → 무효. 피벗 = 베이스 기간 고가. 베이스 스테이지 카운트: 1~3단계 허용, 4단계 이상 금지, 직전 베이스 저점 이탈 시 카운트 리셋. 제외 패턴(하이타이트 플래그, 어센딩 베이스, 스퀘어 박스, IPO 베이스)은 구현 대상이 아니다.
3. **품질 조건**: 돌파 전 10일 레인지 ≤ 피벗의 10% (VCP 수축), 돌파 전 10일 평균 거래량 < 베이스 기간 일평균 거래량 (드라이업)
4. **과열 제외**: 20일 수익률 > 50% 시 진입 금지
5. **상대강도(RS)**: 종목의 상장시장에 맞는 벤치마크(코스피/코스닥) 대비 RS 계산
6. **진입·피라미딩**: 돌파일 1차 진입 → 당일 거래량 검증 → 2·3차 트랜치는 개별 트리거가와 트랜치당 +3% 상한으로 예약 주문
7. **손절**: 2×ATR(14), -10% 캡
8. **청산**: 8주 룰, 60일선 이탈 등 규칙서에 명시된 청산 규칙 전부
9. **포지션 사이징**: 비중 = 1 ÷ 손절% × 100 (스탑아웃당 계좌 손실 ≈ 1%)
10. **포트폴리오 제약**: 동시 보유 최대 8종목, 신규 진입 시 현금·슬롯 확인

## 4. 실행 모드

- **단일 종목 모드**: 종목 CSV 1개 입력 → 날짜별 규칙 판정 로그와 트레이드 결과 출력 (규칙 검증용)
- **포트폴리오 모드**: 종목 CSV 디렉토리 입력 → 사이징, 8종목 상한, 현금 관리를 포함한 전체 시뮬레이션

## 5. 입력 데이터 사양

### 5.1 종목 일봉 CSV (종목당 1파일)

| 컬럼 | 필수 여부 | 용도 |
|---|---|---|
| date | 필수 | YYYY-MM-DD |
| open, high, low, close | 필수 | 수정주가 기준이어야 함 |
| volume | **필수** | 돌파일 거래량 검증, 드라이업 조건, 20일 거래량 |
| value (거래대금) | 선택 | 유동성 하한 필터 대용 |

### 5.2 종목 메타데이터

- **시장구분 (KOSPI / KOSDAQ)**: RS 벤치마크 매칭에 필수. 파일명 규칙, 별도 메타 CSV, 컬럼 추가 중 어떤 방식으로 받을지 계획서에서 결정하고 명시하라.
- (선택) 시가총액 또는 상장주식수: 유동성 하한 필터용

### 5.3 레퍼런스 CSV 2개 — 코스피 지수, 코스닥 지수 일봉

- date, close 필수 (OHLC가 함께 있어도 무방)
- 용도: RS 계산, 거래일 캘린더 기준

### 5.4 데이터 요건과 검증

- **워밍업 구간**: 백테스트 시작일 이전 최소 약 300거래일(52주 신고가 + 200일선 + RS 산출 여유) 데이터가 필요하다. 부족한 종목은 판정 시작일을 자동으로 뒤로 미루는 로직을 포함하라.
- 거래일 캘린더는 지수 CSV의 날짜를 기준으로 하고, 종목별 결측일(거래정지 등) 처리 방침을 정의하라.
- 로더에 검증 로직 포함: 컬럼 존재, 날짜 정렬·중복, high ≥ low, 결측치, 인코딩(UTF-8/CP949).
- 데이터 주의사항으로 기록: 수정주가 여부 확인 방법, 유니버스 구성 시 생존 편향(상장폐지 종목 포함 여부).

## 6. 출력 사양

- **트레이드 로그 CSV**: 종목, 트랜치 번호, 진입/청산 일자·체결가·수량, 진입/청산 사유(규칙 ID), 개별 손익, 비용
- **일별 자본곡선 CSV**: 현금, 평가액, 보유 종목 수
- **성과 리포트**: 총수익률, CAGR, MDD, 승률, 손익비, 기대값(R 기준), 평균 보유기간, 평균 노출도, 총 거래비용
- **육안 검증용 이벤트 목록**: 돌파 후보 발생일 목록 — 자동 필터가 V자 회복과 정상 조정을 완전히 구분하지 못하므로, 차트 확인 워크플로에 연결할 수 있게 한다

## 7. 아키텍처 요구사항 (필수 준수)

1. **클래스 1개 = 책임(데이터) 1개 = 파일 1개**
2. **의존성 주입**: 다른 클래스의 데이터가 필요하면 생성자에서 주입한다. 추상 인터페이스(ABC 또는 typing.Protocol)를 먼저 정의해 데이터 소스와 규칙 구현을 나중에 교체할 수 있게 한다. (예: CsvDataSource → 이후 API 소스로 교체)
3. **파라미터 외부화**: 규칙의 모든 수치(15%, 33%, 5주, 7주, 10%, 2×ATR, -10%, 8종목 등)는 YAML 또는 JSON 설정 파일로 분리하고, 설정에 규칙서 버전(v3.2)을 태깅해 결과 재현성을 확보한다. 코드 내 수치 하드코딩 금지.
4. **단계별 독립성**: 단계 간 계약은 인터페이스로 고정하고, 각 단계는 자체 유닛테스트를 남긴다.

이 구조를 기본으로 하되, 더 나은 설계(레이어 분리 방식, 일별 이벤트 루프 vs 벡터화 등)가 있으면 **대안과 트레이드오프를 함께 제시**하라. 임의로 대체하지 말 것.

## 8. 백테스트 정합성 요구사항 (필수 준수)

- **룩어헤드 편향 금지**: 각 규칙의 판정 시점(D일 종가 확정 후)과 체결 시점을 명시적으로 분리하라.
- **일봉 기반 체결 모델 명문화**: 일봉 OHLC만으로 장중 체결을 시뮬레이션하므로, 다음 가정을 계획서에 표로 명시하라.
  - 돌파 1차 체결가: 예) 피벗이 당일 저가~고가 범위 안이면 max(시가, 피벗)에 체결, 갭 상승 시 처리 방법
  - 피라미딩 트리거 체결과 트랜치 +3% 상한 초과 시 미체결 처리
  - 손절 체결가: 저가 < 손절가이면 손절가 체결, 갭 하락으로 시가 < 손절가이면 시가 체결
  - 돌파일 거래량 검증 실패 시 처리 (규칙서 확인 후 반영)
  - 동일 날짜에 신호 다수 발생 시 우선순위
- **거래비용**: 수수료, 거래세, 슬리피지를 파라미터로 분리한다. 거래세는 연도별 변동 이력이 있으므로 기간별 설정이 가능한 구조를 권장한다.
- **결정론**: 동일 입력 + 동일 설정 → 항상 동일 결과.

## 9. 테스트 요구사항

- 프레임워크: pytest. 단계마다 유닛테스트 필수.
- 합성 데이터 픽스처로 경계 사례 검증: 깊이 15%/33% 경계, 5주/7주 경계, 스테이지 리셋, 과열 50% 경계, 갭 손절, ATR 캡 발동, 8종목 상한 도달.
- 통합 단계에서 소형 실데이터 스모크 테스트 1건.

## 10. 참고 로드맵 초안 (재구성 가능 — 더 나은 분해가 있으면 근거와 함께 수정 제안)

- Phase 0: 프로젝트 골격, 설정 로더, 거래일 캘린더, CSV 로더 + 검증
- Phase 1: 지표 클래스 (이동평균, ATR, 52주 고저, RS)
- Phase 2: 셋업 필터 (추세 템플릿, 과열 제외, RS 조건)
- Phase 3: 베이스 판별기 (깊이 티어, 기간, 스테이지 카운트, 품질 조건)
- Phase 4: 체결 시뮬레이터 (진입·피라미딩·손절·청산 체결 모델, 비용 반영)
- Phase 5: 포지션 사이저 + 포트폴리오 매니저 (8종목 상한, 현금 관리)
- Phase 6: 백테스트 엔진 (일별 이벤트 루프, 전체 컴포넌트 조립)
- Phase 7: 리포팅 (트레이드 로그, 자본곡선, 성과 지표, 이벤트 목록)
- Phase 8: 통합 검증, 회귀 테스트, 사용 문서

## 11. 계획서에 반드시 포함할 항목

1. 전체 아키텍처 개요, 디렉토리 구조, 텍스트 클래스 다이어그램 (의존 방향 표시)
2. 핵심 인터페이스 시그니처 (의사코드)
3. 단계(세션)별 상세: 목표 / 산출 파일 / 구현 클래스 / 입출력 계약 / 테스트 항목 / 완료 기준(DoD) / 1세션 내 완료 가능 여부
4. 체결 가정·엣지 케이스 표와 각각의 처리 방침 (§8 기반)
5. 범위 제외와 후속 과제: 예) 피어그룹 내 RS 랭크(대장주 판별)는 종목 유니버스 데이터가 필요하다 — v1 포함 여부와 필요 데이터를 판단해 제안하라
6. **결정 필요 질문 목록**: 규칙서와 모호하거나 충돌하는 지점은 임의로 가정하지 말고 질문으로 남겨라

## 12. 진행 규칙

- 지금 세션에서는 계획서만 작성한다.
- 계획서 승인 후, 이후 세션들에서 Phase 단위로 구현한다. 각 구현 세션을 시작할 때 붙여넣을 최소 컨텍스트(해당 Phase의 인터페이스 계약, 선행 산출물 목록)를 계획서 안에 Phase별로 미리 정의해 두어라.
